　本プロジェクトは、保育園向け納品業務で使用していたWindowsタブレット+汎用サーマルプリンタを中心の現場端末をiPhone+BLE対応サーマルプリンタへ構成を移行した業務端末・出力インフラの標準化である。従来は中華系8インチのWindows端末を採用していたが、不安定な供給、落下による故障、耐久性の低さ、ドライバ依存による印刷不安定などの問題が慢性的に発生していた。端末故障時には代替機種の選定・検証・再テストが必要となり、現場運用に継続的な負荷を与えていた。

本再構築の目的は、①機器標準化による保守性の向上、②現場負荷の軽減、③印刷処理の高速化・安定化、④API通信の最適化によるサーバ負荷低減である。当初はPWAでの運用を検討したが、SafariのBlueTooth制約により実現が難しく、Apple Developer Enterprise Programを活用した社内配布型アプリへ方針変換した。構成はNext.js+Pleasanter APIを基盤とし、WKWebView経由でBLEプリンタSDKと連携する方式を採用、MDM（Microsoft Intune）で端末配布・利用者設定を統制した。2025年11月より運用を開始し、12月に本格運用へ移行。通信料は約200KBから1.2KBへ削減し、印刷は登録操作からやく2秒以内でかんりょうするなど、性能と安定性の向上を実現した。

従来のWindowsタブレット＋汎用サーマルプリンタ環境は、単なる端末選定の問題ではなく、「継続運用の不安定さ」が根本課題であった。端末は主に中華系製品に依存しており、供給終了や品質ばらつき、落下故障が頻発していた。その都度代替機種の選定・検証を繰り返す必要があり、運用負荷が蓄積していた。私はこの構造を一時的対処ではなく、端末基盤そのものの見直しが必要な状況と判断した。

当初はPWA構成での再設計を検討した。ブラウザ中心であれば柔軟に更新できると考えたためである。しかしSafariではBluetooth APIが利用できず、BLEプリンタとの直接連携が不可能であることが判明した。そこでApple Developer Enterprise Programを活用した社内配布型アプリへ方針転換を行った。iOSやXcode、SDKは未経験領域であったが、調査と検証を重ね、WKWebViewを利用した構成で設計を進めた。ネイティブ全面実装ではなく、Web技術を軸に構築することで、開発効率と将来的な改修容易性の両立を図った。

設計段階で重視したのは安定性と通信効率である。既存APIは約200KBの通信量を伴っており、モバイル環境では遅延やサーバ負荷が懸念された。不要データを削減し、必要最小限のみを取得する設計へ変更した結果、通信量は約1.2KBまで削減された。また試験運用中に納品価格が0円になる不具合が発生した。非同期処理の順序保証不足やAPI取得失敗が影響している可能性を想定し、Promise制御の整理、処理キュー化、API最大3回再試行、取得失敗時の即時エラー通知（StepTask連携およびSlack通知）を実装した。結果として再発は確認されていない。

ユーザー視点では、端末はiPhone＋約180gのプリンタ構成となり、大幅に軽量化された。到着時に最寄り保育園が自動選択され、登録後約3秒以内に印刷が完了する仕組みにより、作業負荷は確実に軽減された。実地確認時にも操作性および速度面で好意的な評価を得ている。

全体最適の観点では、iPhone SEシリーズへの統一、プリンタ機種標準化、MDM（Intune）による端末管理により、将来的な拠点拡張にも対応可能な基盤を整備した。本運用移行後は重大障害報告はなく、業務基盤として安定稼働している。

コストおよび納期の観点では、既存のWindowsタブレット（約8万円台）とサーマルプリンタ（約1万5千円）の組み合わせは、単価自体よりも故障・再検証に伴う隠れた運用コストが課題であった。代替機種選定、ドライバ検証、再テストといった工程が繰り返されることで、結果として人的コストと時間的損失が発生していた。今回の再構築では、iPhone SEシリーズへの統一およびプリンタ機種標準化により、選定・検証プロセスそのものを簡素化した。また、Apple Developer Enterprise Programの承認およびMDM（Intune）配布体制の整備には一定の準備期間を要したが、試験運用期間を設け段階的に本運用へ移行することで、現場影響を最小化しながら安定導入を実現した。

通信設計の観点では、既存APIのデータ肥大化がサーバ負荷と体感速度に影響していると判断した。実装時に取得データ構造を再整理し、不要項目を削減したことで通信量は約200KBから1.2KBへ大幅に縮小された。これは単なる速度改善ではなく、モバイル環境下における安定性確保および将来的な同時利用増加に備えた負荷分散設計の一環である。また、API取得失敗時の再試行処理（最大3回）を導入し、一時的通信不良への耐性を向上させた。

運用開始後の改善サイクルも設計段階から考慮した。試験運用期間中に発生した納品価格0円表示の不具合に対しては、非同期処理順序保証の再設計、処理キュー化、取得失敗時の即時エラー登録機能を実装した。エラー情報はStepTaskへ自動記録され、Mattermostへ通知される仕組みとし、障害発生時の初動対応速度を高めた。本運用移行後は重大障害報告はなく、問い合わせ件数も発生していない。これは構築段階で再発防止を前提とした設計が功を奏した結果であると考えている
